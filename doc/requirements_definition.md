# TwitchFollowerChecker 要件定義書

本ドキュメントは、Qt6 C++で開発するTwitchフォロワーチェッカーツール（TwitchFollowerChecker）の要件を定義します。

---

## 1. プロジェクト概要
Twitch配信活動を行う配信者が、自身のチャンネルの「フォローしているユーザー（Following）」と「フォローされているユーザー（Followers）」を自動取得し、その差異や関係性（相互フォロー等）を簡単に視覚化・データ出力できるデスクトップアプリケーションです。

特に、ストリーマーがOAuth等の面倒なAPI設定で挫折することなく、スムーズに利用開始できる「ユーザーフレンドリーな設計」を重視します。

---

## 2. 動作環境・システム前提
* **対象OS**: Windows 10/11（Qt6 を採用するため、将来的なマルチプラットフォーム化も考慮したコード構成とする）
* **開発言語/フレームワーク**: C++17 / Qt 6.x / CMake 3.16+
* **使用する外部API**: Twitch Helix API

---

## 3. 機能要件

### 3.1. OAuth認証機能（必須・基盤）
ユーザー本人のチャンネル情報を取得するため、安全かつ簡易な認可フローを提供します。
* **認証フロー方式**: **ブラウザ自動連携方式（ループバックサーバー方式）**
  * アプリ側で一時的にローカルHTTPサーバー（例：`http://localhost:8080`）を起動します。
  * 「認証」ボタンクリックで、PCの既定のブラウザでTwitchの認証画面を開きます。
  * ユーザーが認可すると、ブラウザがローカルHTTPサーバーにリダイレクトし、アプリが自動でアクセストークンをキャッチします。
* **取得スコープ**:
  * `user:read:follows` （フォローしているユーザー情報の取得に必要）
  * `moderator:read:followers` （フォロワー情報の取得に必要）
* **対象チャンネル**: ログインした認証ユーザー本人のチャンネル

### 3.2. フォロー中一覧取得機能（設定でON/OFF可能）
ログインユーザーがフォローしている配信者の一覧を取得します。
* **APIエンドポイント**: `/helix/channels/followed`
* **取得データ**: ユーザーID、ログイン名（英数字）、表示名（日本語など）、フォローした日時

### 3.3. フォロワー一覧取得機能（設定でON/OFF可能）
ログインユーザーのチャンネルをフォローしているリスナーの一覧を取得します。
* **APIエンドポイント**: `/helix/channels/followers`
* **取得データ**: ユーザーID、ログイン名（英数字）、表示名（日本語など）、フォローされた日時

### 3.4. フォロー vs フォロワー比較機能（設定でON/OFF可能）
取得した「フォロー一覧」と「フォロワー一覧」を照合し、関係性を分類します。
* **前提条件**: 「フォロー中一覧取得」および「フォロワー一覧取得」の両方がONであること。
* **分類カテゴリ**:
  1. **相互フォロー**: 双方がフォローし合っている状態
  2. **片想い（Followingのみ）**: 自分はフォローしているが、相手からはフォローされていない状態
  3. **片思われ（Followerのみ）**: 相手からフォローされているが、自分はフォローしていない状態

### 3.5. CSV出力機能
取得したデータの一覧をCSV形式でファイル出力します。
* **出力契機**: 画面内のボタン押下時に「名前を付けて保存」のファイル選択ダイアログ（`QFileDialog`）を起動し、ユーザーが指定した保存先・ファイル名で出力する。
* **出力項目**:
  * ユーザーID
  * ログイン名（ID）
  * 表示名（Display Name）
  * フォロー日時（またはフォローされた日時）
  * **関係状態（相互、片想い、片思われ等のステータス記載）**

### 3.6. API資格情報（Client ID / Secret）の管理
セキュリティと利便性を両立するため、ハイブリッドな資格情報管理を行います。
* **デフォルト認証情報の埋め込み**:
  * 開発者（BLUE000氏）が用意した Client ID / Secret をあらかじめビルド時に埋め込みます。
  * セキュリティ担保のため、これらの認証情報はGit管理外のローカルファイル（`local_config.cmake`）に定義し、CMake経由でビルド時のみコードに展開します（ソースコード上に直書きして公開しない）。
* **カスタム認証情報の上書き設定**:
  * 上級ユーザーが自身の Client ID / Secret を使用して動作させたい場合、アプリの設定画面からカスタムの `Client ID` / `Secret` を入力して上書きできる仕組みを提供します。

### 3.7. 各種設定の保存機能（一部検討中）
* 各種機能（フォロー取得、フォロワー取得、比較）のON/OFF状態や、カスタムAPI資格情報を保持します。
* **画面UI**: 専用の「設定画面」（または設定タブ）内にチェックボックス等を配置して制御します。
* **保存方法**: **【現在 検討中】**（次のステップにて設計）

## 4. 非機能要件
* **セキュリティ**: 
  * TwitchアクセストークンやAPIシークレットは、絶対にログに出力しないこと。
  * Gitの公開リポジトリに開発用の Client Secret を混入させないビルド構造にすること。
* **ユーザビリティ**: 
  * 配信者が一目見て直感的に設定変更や取得実行ができるよう、わかりやすい日本語UIとシンプルな動線にする。
* **エラーハンドリング**: 
  * Twitch APIのレートリミット（アクセス制限）や、ネットワーク切断時の適切なタイムアウト・エラー表示を行うこと。

---

## 5. ユースケース（主な操作動線）
1. アプリを起動し、**「設定」タブ/ダイアログ**で必要な機能（フォロー/フォロワー取得、比較）をONにする。
2. メイン画面の「ログイン/認証」ボタンを押し、ブラウザ経由でワンクリックで連携を完了させる。
3. 「実行」ボタンを押し、選択した機能の実行結果（リスト）を画面上にグリッド表示する。
4. 「CSVエクスポート」ボタンを押し、ダイアログで指定したパスに結果をCSV保存する。
